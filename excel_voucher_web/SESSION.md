# SESSION.md

## 운영 원칙
- 컨텍스트 절약 최우선.
- 긴 분석, 로그, 결정사항, 진행상황은 채팅에 누적하지 말고 MD 파일에 갱신한다.
- 작업 시작 전 관련 MD와 graphify 결과를 먼저 읽는다.
- 필요한 파일만 좁게 열어 진행한다.
- 코드 구조, 의존관계, 영향범위 확인은 graphify를 적극 활용한다.
- 답변은 핵심만 짧게 한다.
- 자세한 인수인계 정보는 MD에 남긴다.
- 작동하던 기존 기능은 임의 삭제 금지.
- 삭제가 필요하면 먼저 질문한다.

## 신규 목표
재정부서 회계업무 자동화 확대 전개.
세금계산서 메일 수신 기반이 아니라, 담당자가 엑셀을 업로드하면 전표 처리를 수행하는 기능을 개발한다.

## 기본 흐름
1. 담당자가 엑셀 업로드.
2. 업로드는 드래그 앤 드롭 또는 파일 선택으로 가능.
3. 업로드 시 회계일 입력/수정 가능.
4. 서버는 작업을 큐에 등록.
5. 담당자는 창을 닫아도 됨.
6. 172.17.30.243 자동 전표처리 PC Agent가 자동으로 ERP 전표 작성/출력 수행.
7. 출력 제출 성공 후 담당자에게 완료 메일 발송.

## 계정/메일 방향
- 그룹웨어 MariaDB `ksystem_yundong`의 `gw_emp`, `ds_t_emp`에서 담당자 계정/메일/부서 정보를 읽는다.
- 외부 DB는 읽기 전용으로만 사용한다.
- 로컬 시스템 초기 비밀번호는 `wowjd12!@`.
- 최초 로그인 및 임시 비밀번호 로그인 시 비밀번호 변경을 강제한다.

## 이번 세션 구현 범위
- 기존 `web_v1` 기능은 수정하지 않는다.
- 새 시스템은 `excel_voucher_web/`에 독립 구성한다.
- 1차 구현은 엑셀 업로드, 전표 payload 생성, 서버 작업 큐, Agent 폴링 API, 담당자용 웹 화면까지 포함한다.
- 실제 ERP 화면 자동입력은 `app/agent_adapter.py`와 `agent/agent_worker.py`에 분리된 후속 구현 지점으로 둔다.

## 2026-07-20 현재 운영 상태
- ERP 하단 관리항목은 화면에 보이면서 UIA에는 `labels=[]`가 되는 GDI 화면이다. 준비 완료는 UIA 글자가 아니라 관리항목 빨간 라벨과 실제 수평 구분선의 안정된 화면 signature 2회로 판정한다.
- 첫 미지급금은 관리항목값 1회 클릭 → F9 → 거래처번호 → Tab 4 → Down 5 → Up 2 → Tab 3 → Enter 2이며, 후속 미지급금은 1회 클릭 → 거래처번호 → Enter다.
- 한 화면의 마지막 완전 표시 행과 전체 행 수는 고정하지 않는다. 실제 구분선 Y로 24/26 등 현재 표시행을 계산하고 payload의 마지막 보통예금까지 같은 경계 상태기계를 사용한다.
- 마지막 보통예금은 최초 F9 성공 좌표를 계좌번호/금융기관지점에 재사용한다. 운영 fast 경로에서는 관리항목명/계정과목 확인을 위한 전체 UIA와 GDI Ctrl+C 검증을 하지 않는다.
- 최종 private 회귀 테스트는 93 passed 상태다. 다음 실기기 실행에서는 로그의 `GDI 하단 관리항목 화면 안정 감지` 직후 1행 F9가 시작되는지 확인한다.

## 2026-07-21 현재 운영 수정
- 1행 진입 전 멈춤의 직접 원인은 화면 준비 감지 이후 실행된 `Desktop.from_point()` 기반 보이는 행 재검사였다. 운영 fast 경로에서는 이 point UIA와 전체 UIA 행 탐색을 호출하지 않고, 이미 검증된 GDI 화면 경계로 만든 1..24/26 좌표를 바로 사용한다.
- 첫 행은 검증된 기존 Y 좌표 `231`을 유지한다. 임시로 검토했던 `240`은 24행 화면을 23행으로 계산할 수 있어 반영하지 않았다.
- 첫 미지급금 행은 관리항목값 1회 클릭 → F9 → 최소 1초 화면 전환 대기 → 거래처번호 → Tab 4 → Down 5 → Up 2 → Tab 3 → Enter 2 순서다. 운영 fast 경로에서는 F9 뒤 UIA 창/자손 열거를 하지 않는다.
- 화면 경계를 지난 뒤에는 현재 화면의 마지막 완전 표시 행 Y를 고정한다. 거래처번호 Enter가 다음 행 이동을 수행한 경우 같은 Y를 재사용하고, Enter가 없는 행만 Down 1회를 보낸다. 총행은 payload 값이므로 230/300행과 마지막 동적 보통예금 행까지 같은 방식으로 진행한다.
- 마지막 보통예금의 기존 계좌번호/금융기관지점 입력 분기는 변경하지 않았다. 전체 회귀 테스트는 95 passed 상태다.
